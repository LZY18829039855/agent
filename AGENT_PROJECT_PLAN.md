# Agent 项目搭建规划

## 一、项目目标与范围

- **目标**：基于你已有的「模型接口」和「各种工具接口」，实现一个能自主决策、调用工具的 Agent。
- **范围**：Agent 核心循环（模型 + 工具调用）、可扩展的工具注册、清晰的接口约定。

---

## 二、整体架构（高层）

```
用户输入
    ↓
┌─────────────────────────────────────────────────────────┐
│                    Agent 主循环                          │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐   │
│  │ 模型接口层   │ → │ 输出解析层   │ → │ 工具执行层   │   │
│  │ (你的API)   │   │ (tool call) │   │ (你的工具)   │   │
│  └─────────────┘   └─────────────┘   └─────────────┘   │
│         ↑                                    │          │
│         └────────── 下一轮输入 ◄──────────────┘          │
└─────────────────────────────────────────────────────────┘
    ↓
最终回复 / 中间结果
```

## 三、分阶段流程

### 阶段 1：需求与约定整理（1–2 天）

| 步骤 | 内容 | 产出 |
|------|------|------|
| 1.1 | 明确模型接口形式 | 请求/响应格式（含是否支持 function calling / tool use） |
| 1.2 | 盘点所有工具接口 | 每个工具：入参、出参、调用方式（HTTP/gRPC/本地函数等） |
| 1.3 | 确定技术栈 | 语言（如 Python）、是否用现有框架（LangChain、LlamaIndex、自研等） |
| 1.4 | 约定项目结构 | 如 `agent/`、`tools/`、`models/`、`config/` 等目录与职责 |

**建议**：把「模型接口文档」和「工具接口列表」各整理成一份简短说明（可放在 `docs/` 或 README），便于后续实现时对照。

---

### 阶段 2：模型接口适配层（2–3 天）

| 步骤 | 内容 | 产出 |
|------|------|------|
| 2.1 | 封装模型调用 | 统一函数：`chat(messages, tools=None)`，内部调用你的模型 API |
| 2.2 | 支持「工具描述」传入 | 若模型支持 function calling，将工具 schema 转为模型要求的格式并随请求发送 |
| 2.3 | 统一响应结构 | 解析模型返回，得到：`content`（文本）和/或 `tool_calls`（名称 + 参数） |
| 2.4 | 处理流式/非流式 | 若需要流式输出，在本层做流式解析与 tool call 聚合 |

**注意**：若你的模型不支持 tool use，需在「输出解析层」用规则或小模型从自然语言中解析出「要调用的工具 + 参数」。

---

### 阶段 3：工具层设计与实现（2–3 天）

| 步骤 | 内容 | 产出 |
|------|------|------|
| 3.1 | 定义工具 Schema | 每个工具：`name`、`description`、`parameters`（JSON Schema），供模型理解与生成调用 |
| 3.2 | 实现工具执行器 | 根据 `(tool_name, arguments)` 调用对应工具接口，捕获异常、超时 |
| 3.3 | 工具注册机制 | 支持从配置文件或代码中注册/启用部分工具，便于扩展与开关 |
| 3.4 | 统一返回格式 | 工具结果统一为「可被拼回对话」的形式（如字符串或结构化摘要），便于作为下一轮模型输入 |

**建议**：工具名、参数名与模型侧、文档中保持一致，避免歧义。

---

### 阶段 4：Agent 主循环（3–4 天）

| 步骤 | 内容 | 产出 |
|------|------|------|
| 4.1 | 设计对话状态 | 维护 `messages` 列表（user / assistant / tool 等角色），支持多轮 tool call |
| 4.2 | 实现主循环逻辑 | 循环：调用模型 → 若无 tool_calls 则结束并返回；若有则执行工具、将结果写入 messages、再调模型 |
| 4.3 | 终止条件 | 最大步数、超时、用户中断、模型明确「最终回答」等 |
| 4.4 | 错误与重试 | 工具失败时的重试策略、是否将错误信息反馈给模型让其重试或换策略 |
| 4.5 | Prompt 设计 | 系统提示词：说明 Agent 身份、可用工具、输出格式（如必须用指定格式返回 tool call） |

**注意**：若模型原生支持 function calling，则 4.5 主要约束「行为」；若不支持，Prompt 中需约定如何用自然语言或结构化文本表达工具调用，并在解析层实现对应解析。

---

### 阶段 5：配置、安全与可观测（1–2 天）

| 步骤 | 内容 | 产出 |
|------|------|------|
| 5.1 | 配置外部化 | API Key、模型名、工具开关、最大步数等放入配置文件或环境变量 |
| 5.2 | 安全 | 不在代码中硬编码密钥；对工具入参做校验，避免注入或越权调用 |
| 5.3 | 日志与追踪 | 记录每轮请求/响应、工具调用与结果，便于调试与审计 |
| 5.4 | 可选：简单监控 | 统计调用次数、延迟、失败率等，便于后续优化 |

---

### 阶段 6：测试与迭代（持续）

| 步骤 | 内容 | 产出 |
|------|------|------|
| 6.1 | 单测 | 模型适配层 mock 测试；工具执行器单测；主循环用 mock 模型/工具做逻辑测试 |
| 6.2 | 端到端用例 | 用真实模型 + 真实工具跑若干典型场景，验证「能正确选工具、传参、多轮、收尾」 |
| 6.3 | 边界与异常 | 工具超时、模型返回格式异常、无可用工具时的行为等 |
| 6.4 | 迭代 | 根据表现调整 Prompt、工具描述、解析规则或重试策略 |

## 四、推荐项目目录结构（示例）

```text
agent_project/
├── README.md
├── config/
│   ├── agent.yaml          # agent 行为、最大步数等
│   └── tools.yaml          # 工具开关、端点等
├── src/
│   ├── agent/
│   │   ├── __init__.py
│   │   ├── loop.py         # 主循环
│   │   ├── prompts.py      # 系统提示词与模板
│   │   └── state.py        # 对话/会话状态
│   ├── models/
│   │   ├── __init__.py
│   │   └── adapter.py      # 你的模型接口封装
│   └── tools/
│       ├── __init__.py
│       ├── registry.py     # 工具注册与发现
│       ├── executor.py     # 工具执行
│       └── definitions/    # 各工具的 schema 与实现
│           ├── tool_a.py

├── tests/
│   ├── test_agent_loop.py
│   ├── test_tools.py
│   └── test_model_adapter.py
├── requirements.txt
└── docs/
    ├── model_api.md        # 模型接口说明
    └── tools_api.md        # 工具接口说明
```

你可按实际技术栈（如 TypeScript、Go）调整命名与层级。

## 五、关键决策清单（建议先定下来）

在动手写代码前，建议先明确：

1. **模型是否支持 function calling / tool use？**  
   - 是 → 重点做 schema 对齐与响应解析。  
   - 否 → 需要设计「自然语言/结构化文本 → 工具调用」的解析方案。

2. **工具调用方式？**  
   - HTTP、gRPC、本地 Python 函数等 → 决定执行器如何调用、如何做超时与错误处理。

3. **是否要多轮对话/持久化？**  
   - 仅单次任务 vs 多轮会话 → 影响是否要会话 ID、历史存储与上下文裁剪策略。

4. **是否使用现成框架？**  
   - LangChain/LlamaIndex 等可加快开发，但需把你的模型接口和工具「适配」到框架；自研则灵活度更高、依赖更少。

## 六、时间与优先级建议

- **最小可用版本（MVP）**：阶段 1 + 2 + 3 + 4，即：模型适配 + 工具注册与执行 + 主循环 + 基础 Prompt。先跑通一条端到端链路。
- **再补**：阶段 5（配置、安全、日志）和阶段 6（测试与用例迭代）。
- 若时间紧，可先实现 1 个模型 + 2–3 个工具，验证流程后再扩展工具和策略。
